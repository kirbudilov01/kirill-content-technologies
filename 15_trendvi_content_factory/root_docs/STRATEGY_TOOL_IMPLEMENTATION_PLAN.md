## SERVER REQUIRED (делать только когда готово)

### Нужны ли отдельные воркеры?

Короткий ответ: **да, лучше разнести на 2 воркера**.

1. `strategy_recommender_worker`
	- считает консультационный summary/one-pager,
	- ранжирует top actions,
	- сохраняет рекомендации с версией формулы.

2. `demand_gap_worker`
	- считает спрос/предложение по темам,
	- обновляет opportunity score,
	- прогревает витрины для режима `Underserved`.

Почему это лучше:
- не блокирует API и UI,
- можно пересчитывать батчами,
- проще контролировать качество и откаты,
- независимые релизы и SLA по джобам.

Текущий статус:
- endpoint и Celery task names уже добавлены,
- отдельные docker-compose worker сервисы добавлены,
- запуск и операционное включение выполняется на сервере.

### A) Качество авто-классификатора контента

**Зачем:** снизить ошибки rule-based классификации, добавить confidence.

**Что добавить:**
- поле `content_type`
- поле `content_type_confidence`
- поле `content_type_source` (`rule|llm`)
- воркер для перерасчета классификации
- endpoint для чтения confidence/источника в карточках

**Команды (примерный шаблон, запускать вручную):**
- миграции БД
- перезапуск воркеров celery
- бэкофисовый прогон backfill

### B) Long-tail 90–180 дней

**Зачем:** реальная метрика устойчивого трафика.

**Что добавить:**
- таблица `video_snapshots_daily`
- расписание сбора снапшотов
- агрегация long-tail score (90/180)

**Команды (примерный шаблон, запускать вручную):**
- миграции БД
- запуск/проверка scheduler
- перезапуск воркеров

### C) Недозаполненный спрос v2

**Зачем:** не только прокси по формату, а реальный demand/supply сигнал.

**Что добавить:**
- источник поискового спроса (или внутренние прокси по темам)
- topic-level demand table
- воркер пересчета opportunity score
- endpoint `strategy_meta` для статусов (`strategy_status`, `demand_status`, `classifier_status`, `freshness_status`)
