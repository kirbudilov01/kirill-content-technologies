# AtlasRepo YouTube Content

## Core Output

Produce practical creator assets, not abstract strategy:

1. Long-form video concepts and read-aloud scripts.
2. Shorts scripts that can be read directly from a teleprompter.
3. Repo/topic selection with clear “why viewers care.”
4. Partner screencast briefs: exact GitHub/product/demo screens to record.
5. Native AtlasRepo mentions placed inside the content flow.
6. Funnel CTA: point viewers to the free AtlasRepo scanner for open-source projects/AI solutions via the channel header.
7. Optional DOCX/Google Docs-ready deliverables when requested.

Default language follows the user. If the user asks for US English, write native, simple, creator-friendly American English.

## Content Philosophy

Use context-first storytelling:

Bad:
“Postiz is an open-source social media scheduling tool.”

Good:
“Recording one video is no longer the hard part. Getting it everywhere is. Postiz is an open-source social media scheduler that can help plan, schedule, and distribute content across platforms.”

For every repo or tool, write in this order:

1. Normal-person problem or desire.
2. Repo/tool as the surprising answer.
3. What the viewer sees on screen.
4. Why it matters now.
5. Distinct closing beat.

Avoid starting with technical labels unless the audience already needs them. Prefer “what job does this replace?” over “what category is this?”

## Topic Selection

Prioritize repos and themes that feel:

- On the audience’s radar or likely to be shared.
- Immediately testable or visually demonstrable.
- Useful to creators, founders, developers, marketers, indie hackers, or vibe coders.
- Non-duplicative inside the same batch.
- Strong enough to make the viewer think: “I need to save this channel.”

For repo batches, avoid ten variations of the same tool. Build a stack of roles:

- Browser worker
- Video generator
- Local AI engine
- AI workspace
- AI app builder
- Visual workflow
- Coding worker
- Agent/team manager
- CRM/business layer
- Distribution/publishing layer

When choosing current repos, verify GitHub URLs and freshness because stars, repo names, popularity, and project status change.

## Long-Form Rules

Target long videos are usually 10-20 minutes. A useful draft should be 1000+ words when the user asks for scripts, not just bullet points.

The first 30 seconds must be broad and obvious. Do not open with niche repo jargon. Open with a high-stakes situation:

- “Imagine trying to build a one-person AI company in 2026.”
- “Most creators do not have an idea problem. They have a distribution problem.”
- “I wanted to see if GitHub could replace parts of a content team.”

Use a three-act feel without labeling it:

1. Act 1: Hook, promise, stakes, map.
2. Act 2: Concrete repo/tool demos, escalating curiosity, honest caveats.
3. Act 3: Synthesis, AtlasRepo method, practical next step.

Place AtlasRepo mentions naturally:

- Around 30%: explain how the projects were found or filtered.
- Around 70%: explain the repeatable method and why AtlasRepo is the radar/map.

Never make AtlasRepo feel like a hard ad break. It should sound like the discovery engine behind the episode.

When the goal is traffic to AtlasRepo, include a direct but natural CTA:

- “We found these while scanning open-source projects in AtlasRepo.”
- “If you want to scan this kind of stack yourself, the free AtlasRepo scanner is linked in the channel header.”
- “The free scanner for open-source AI tools and projects is in the channel header.”

## Shorts Rules

Shorts should be 30-60 seconds and easy to read aloud. Aim for 65-95 English words unless the user asks otherwise.

Each Short needs:

1. A broad hook in the first sentence, often “Did you know you can stop paying for...” or “Did you know you can stop doing...”
2. A creator/founder voice line: “I recently found a GitHub repo called...” or “I found an open-source repo that...”
3. One clear repo/tool and the job it replaces.
4. One visual proof request or simple demo image.
5. A comment prompt: “Would you use this?”, “Is this cool or too early?”, “Tell me in the comments.”
6. A CTA to AtlasRepo/free scanner in the channel header.
7. A unique ending.

Preferred Shorts pattern:

“Did you know [normal-person problem / paid tool / manual work]?
I recently found [repo], an open-source project that [main outcome].
It helps with [1-2 concrete uses].
[Balanced caveat or why it matters.]
Would you use this / is this cool / what do you think? Tell me in the comments.
More solutions like this are on AtlasRepo in the channel header.”

Avoid repeated endings like:

- “Verdict: save.”
- “This is worth testing.”
- “Follow for more.”
- Identical “link in bio/header” wording in every Short.

Use CTA wording, but vary it:

- “The free open-source project scanner is linked in the channel header.”
- “If you want to scan more tools like this, the free AtlasRepo link is in the channel header.”
- “We track repos like this in AtlasRepo; the free scanner is in the channel header.”
- “The free scanner for AI tools and GitHub projects is linked from the channel header.”

Use varied closing beats:

- Worker reveal: “automation starts looking like a worker.”
- Uncomfortable question: “test it before it becomes normal.”
- Ownership frame: “you finally own the engine.”
- Bridge frame: “this is the bridge.”
- Speed frame: “open it while the idea is still warm.”
- Rabbit-hole frame: “real rabbit-hole energy.”
- Review frame: “you still review the diff.”
- Watchlist frame: “keep it on the watchlist.”
- Business frame: “where customers stop getting lost.”
- Distribution frame: “whether anyone sees what you made.”

After drafting Shorts, audit first and last sentences to ensure they do not sound cloned.

## Partner Screencast Briefs

For each repo, give the partner practical capture tasks:

1. GitHub header: owner/repo, stars if useful, short description.
2. README or docs promise.
3. Product website or live app when available.
4. Demo/proof: task, UI, workflow, terminal, output, or generated result.
5. Limitation/risk if setup is hard or output is rough.
6. Horizontal shots for long-form.
7. Vertical crops for Shorts.

Use this screen sequence where possible:

GitHub -> README/docs -> product/demo -> proof/result -> limitation -> clean final visual.

Warn the partner not to show private API keys, tokens, customer data, browser history, or sensitive local paths.

## Script Style

Write scripts as spoken text, not academic analysis.

Prefer:

- “Here is the human version of the problem...”
- “The reason this matters is simple...”
- “I would not overpromise this...”
- “The demo only needs to prove one thing...”

Avoid:

- Overusing “imagine” in every hook.
- Long technical acronyms without context.
- Generic “AI will change everything” claims.
- Repeated “save/test/verdict” endings.
- Dry lists of repo features.

Be honest about rough open-source tools. Trust increases when the script says “this is promising, but the demo/output/setup may still be rough.”

## Audit Before Delivery

Run these checks before finalizing:

1. First 30 seconds: understandable to a tech-curious non-specialist.
2. Repos: non-duplicative and role-based.
3. Shorts: no repeated first-line structure or identical closings.
4. AtlasRepo: included natively, not as a disconnected ad.
5. Shorts and long-form scripts: include a clear AtlasRepo/free scanner CTA when traffic is a goal.
6. Partner brief: enough detail for someone else to record without asking what to show.
7. If generating DOCX, structurally verify it and visually render when the environment supports it.
