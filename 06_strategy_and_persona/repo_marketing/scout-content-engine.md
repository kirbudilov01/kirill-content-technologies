# Scout Content Engine

Date: 2026-06-26

Purpose: define Scout as the core engine of AtlasRepo/FABRICBOT and show how to start with content, research, guides, and promo logic before building the full automated agent factory.

## 1. Core Idea

Scout is not only a discovery job.

Scout is the intelligence core:

- finds tools;
- scores opportunities;
- detects categories;
- feeds workflow packs;
- feeds content;
- feeds SEO;
- feeds guides;
- feeds paid offers;
- feeds product roadmap;
- feeds Codex/Claude implementation tasks.

The marketing system should be built around this:

> Scout finds what matters. AtlasRepo explains what it can become. FABRICBOT/agents help people launch it.

## 2. Product Frame

The product becomes powerful because it combines three layers:

1. Discovery:
   - live repo intelligence;
   - trend signals;
   - category rankings;
   - market watch.

2. Interpretation:
   - what the repo does;
   - who needs it;
   - what workflow it enables;
   - how hard it is to launch;
   - what business use case it supports.

3. Execution:
   - Codex/Claude playbooks;
   - setup guides;
   - prompts;
   - templates;
   - agent automation;
   - community support.

Most competitors stop at discovery or content. The strong position is discovery + interpretation + execution.

## 3. Start With Content

Starting with content is correct because it validates:

- which categories people care about;
- which workflows create desire;
- which offers convert;
- which wording works;
- which repos are visually demoable;
- which problems are urgent.

Before building heavy automation, content should discover the market.

Initial content goal:

> Prove that people want curated AI workflow packs by category.

Do not start with 100 categories. Start with 3.

Recommended first categories:

- Content Engine.
- Lead Generation / Sales Research.
- MVP Builder / AI App Builder.

Why these:

- easy to explain;
- strong business value;
- visual demos are possible;
- Codex/Claude implementation angle is natural;
- they create both creator and founder demand.

## 4. External Guides

External guides are public trust and SEO assets.

They should be useful enough to rank and share, but leave the full implementation layer for paid access.

Guide types:

- `Best open-source AI tools for [category]`
- `How to build a [workflow] with open-source tools`
- `Open-source alternatives to [paid SaaS]`
- `Codex/Claude workflow for [task]`
- `AI agent stack for [audience]`
- `GitHub repos that can become [business system]`

External guide structure:

1. Problem.
2. Best use cases.
3. Recommended repo stack.
4. Comparison table.
5. Setup difficulty.
6. What to watch out for.
7. Mini workflow.
8. CTA to scanner/free workflow list.

External guides should not feel like thin SEO pages. They should feel like field notes from a real scout system.

## 5. Internal Guides

Internal guides are the paid value layer.

They should be operational, not inspirational.

Internal guide types:

- Full workflow pack.
- Codex task sequence.
- Claude planning prompt.
- Setup checklist.
- API integration notes.
- Docker/local deployment notes.
- Data model/template.
- Troubleshooting guide.
- Monetization/use-case guide.
- Content distribution guide for that workflow.

Internal guide structure:

1. Outcome.
2. Required tools/repos.
3. Skill level.
4. Setup path.
5. Codex tasks.
6. Claude prompts.
7. Test checklist.
8. Common errors.
9. How to adapt it.
10. How to sell/use it.

This is where people pay because it saves implementation time.

## 6. Research System

Each research item should become a structured dossier:

```json
{
  "category": "content-engine",
  "problem": "turn one idea into multi-platform content",
  "repos": ["gitroomhq/postiz-app", "n8n-io/n8n"],
  "paid_tools_replaced": ["Buffer", "Zapier", "content calendar tools"],
  "audience": ["creators", "founders", "agencies"],
  "workflow_value": "distribution system",
  "demoability": "high",
  "codex_fit": "medium",
  "claude_fit": "high",
  "seo_angles": [
    "open-source content automation",
    "self-hosted social media scheduler",
    "AI content workflow"
  ],
  "video_angles": [
    "build a content engine from GitHub repos",
    "stop paying for social scheduling tools"
  ],
  "offer_angle": "Content Engine Workflow Pack"
}
```

This dossier can feed every agent later.

## 7. Promo Logic

Every category should have a promo scenario.

### Content Engine Scenario

Pain:

> You have ideas but no distribution system.

Promise:

> Turn one idea into Shorts, long videos, posts, SEO pages, and Telegram updates.

Paid unlock:

- repo stack;
- prompts;
- posting workflow;
- avatar/screencast scripts;
- analytics loop.

### Lead Generation Scenario

Pain:

> You need leads but do not want to manually scrape, enrich, and research prospects.

Promise:

> Build an AI lead research workflow from open-source tools.

Paid unlock:

- sourcing stack;
- enrichment flow;
- outreach research template;
- Codex/Claude setup tasks;
- compliance/risk checklist.

### MVP Builder Scenario

Pain:

> You want to launch but do not know which open-source blocks to start with.

Promise:

> Pick a repo stack and turn it into a working MVP path.

Paid unlock:

- stack map;
- repo setup;
- auth/payments/deploy checklist;
- Codex task breakdown;
- launch content pack.

## 8. Content Formats To Start

Start simple:

- 3 Shorts/day.
- 3-5 X/Threads posts/day.
- 1 Telegram post/day.
- 2 long videos/week.
- 1 external guide/week.
- 1 internal workflow pack/week.

Minimum viable version:

- 1 long-form research per week.
- 1 workflow pack per week.
- 10-15 Shorts from that pack.
- 1 public guide.
- 1 email/newsletter.

The weekly workflow pack is the source asset. Everything else is derivative.

## 9. Weekly Workflow

Monday:

- Scout selects category/theme.
- Research dossier created.

Tuesday:

- Workflow Architect creates stack and guide outline.
- Script Agent creates video/post ideas.

Wednesday:

- First Shorts batch.
- Public guide draft.

Thursday:

- Internal workflow pack draft.
- Codex/Claude implementation tasks.

Friday:

- Long video script or recording.
- Promo post and Telegram update.

Saturday:

- Repurpose into Shorts, threads, newsletter.

Sunday:

- Analytics review.
- Decide next week's category.

## 10. Offer Integration

Each guide and video should map to a paid offer.

Example:

Public guide:

> How to build an open-source content engine.

Free CTA:

> Get the free workflow scanner.

Paid CTA:

> Unlock the full Content Engine Workflow Pack inside AtlasRepo.

Upsell:

> Join the 90-day Builder Sprint to build your first system with Codex/Claude playbooks.

Do not create content without an offer path.

## 11. Why This Can Become Defensible

The edge is not that AtlasRepo has a list of tools.

The edge is the compounding dataset:

- which repos were found;
- which categories they map to;
- which videos performed;
- which SEO pages ranked;
- which CTAs converted;
- which workflows people requested;
- which implementation guides solved problems;
- which Codex/Claude tasks worked.

Over time, this becomes a proprietary intelligence layer around open-source AI workflows.

That is stronger than a course or a static directory.
